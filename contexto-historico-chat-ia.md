## 2. ESTRUTURA MULTI-TENANT E HIERARQUIA DE ACESSO (RBAC)
O sistema opera em um modelo hierárquico estrito de 4 camadas. Toda modelagem de banco e lógica de API deve respeitar este fluxo:

1.  **Usuário Master (Platform Controller/SaaS):** * Controle global da infraestrutura.
    * Cadastra Clínicas (Tenants) e define limites de assinatura (quantos profissionais, pacientes e familiares a clínica pode ter).
2.  **Clínica (Tenant Manager):**
    * Gerencia os Profissionais vinculados a ela.
    * Distribui as cotas do SaaS (ex: define quantos pacientes o Psicólogo X pode atender e limita o registro a 1 ou 2 familiares por paciente).
3.  **Profissional (Terapeuta):**
    * Cadastra os Pacientes (sujeito aos limites da Clínica).
    * Gera códigos de convite (Invite Codes) temporários e criptografados para as Famílias ingressarem na plataforma.
4.  **Família / Paciente:**
    * Acesso restrito apenas via Código de Convite.
    * Após inserir o código no frontend (PWA mobile), o sistema amarra o Auth ID do familiar estritamente ao UUID daquele paciente específico.

## 3. ARQUITETURA DE MICROSSERVIÇOS (SUPABASE EDGE FUNCTIONS + NODE.JS)
O backend não será um monólito. A arquitetura será baseada em **Supabase Edge Functions** rodando lógicas em Node.js/Deno. 

* **Modularidade Obrigatória:** Cada funcionalidade ou domínio lógico (ex: `generate-invite`, `process-audio-ai`, `get-patient-timeline`) deve ser um microsserviço isolado, em seu próprio diretório, com seu próprio arquivo `index.ts`, validações e dependências.
* **Row Level Security (RLS) Primeiro:** No ecossistema Supabase, a segurança da API começa no banco de dados. Qualquer operação de banco de dados (`select`, `insert`, `update`) deve ser protegida por políticas rigorosas de RLS no PostgreSQL. Um Profissional não pode ver pacientes de outro, e um Familiar não pode acessar dados fora do seu escopo, MESMO que a Edge Function seja comprometida.
* **Autenticação e Sessões:** Uso exclusivo do Supabase Auth (GoTrue). A decodificação e validação do JWT nos headers de cada requisição é obrigatória para identificar a `role` e o `tenant_id` da entidade que faz a chamada.

## 4. MELHORES PRÁTICAS DE DESENVOLVIMENTO E SEGURANÇA
Ao escrever o código e as regras de negócio, aplique as seguintes práticas:

* **Validação de Input (Fail-Fast):** Todo payload de entrada nas Edge Functions deve ser validado estritamente utilizando bibliotecas como **Zod** antes de qualquer processamento lógico.
* **Idempotência e Transações:** Requisições críticas (como vinculação de código de convite ou geração de relatórios) devem usar transações SQL (via RPC no Supabase) para evitar estados inconsistentes caso haja falha de rede no meio do processo.
* **Princípio do Menor Privilégio:** As funções devem utilizar o Service Role Key apenas quando estritamente necessário para operações de *bypass* de sistema (ex: cron jobs de faturamento). Operações de usuário devem sempre usar o token JWT do requisitante para herdar as regras de RLS.
* **Padronização de Resposta:** Todas as APIs devem retornar um formato previsível: `{ success: boolean, data?: any, error?: { code: string, message: string } }`.

Design Stateless (Sem Estado): Cada invocação de uma Edge Function deve ser independente. Nenhuma sessão ou dado temporário deve ser armazenado na memória da função, garantindo escalabilidade horizontal infinita sem perda de contexto.

Idempotência em Transações Críticas: Requisições como "Aprovar Relatório" ou "Consumir Crédito de IA" devem ser desenhadas para serem seguras contra retentativas de rede. O uso de chaves de idempotência (Idempotency Keys) no payload evita processamento duplicado.

Assincronicidade e Event-Driven (Webhooks e Triggers): Para processamentos pesados (ex: enviar áudio de 5 minutos para transcrição via IA), a requisição inicial não deve ficar aguardando a resposta (blocking). O frontend é notificado via WebSockets (Supabase Realtime) quando o processamento for concluído.

3. Performance, Resiliência e Banco de Dados

Connection Pooling (PgBouncer): Como estamos usando funções Serverless que escalam rapidamente a zero e a centenas, conexões diretas esgotariam o pool do banco. É mandatório o uso da URL de connection pooling do Supabase para requisições que não utilizem a API REST nativa (PostgREST).

Validação Rígida de Contratos (Fail-Fast): A sanitização e validação de payloads de entrada devem ocorrer no primeiro milissegundo da requisição. O uso de Zod para garantir que a tipagem TypeScript corresponda exatamente ao JSON recebido previne falhas catastróficas e injeções de código.

Rate Limiting e Throttling: Proteção de rotas sensíveis, especialmente as que batem em APIs externas e custosas (como a IA generativa ou rotas de autenticação/convites). Implementar limites de requisição por IP e por tenant_id previne DDoS e custos excessivos.

4. Engenharia e Operações (Observabilidade)

Tracing e Logs Estruturados: Ao invés de usar apenas console.log(), as funções devem emitir logs estruturados em JSON, facilitando a ingestão e busca por sistemas de monitoramento (ex: Datadog, ELK, ou o próprio painel do Supabase). O log deve conter trace_id para acompanhar uma requisição do frontend até o retorno da IA.

Infrastructure as Code (IaC) e Migrations: Qualquer alteração no banco de dados (tabelas, RLS, functions RPC) deve ser feita estritamente através de arquivos de migration do CLI do Supabase. Nenhuma alteração manual em produção é permitida.
